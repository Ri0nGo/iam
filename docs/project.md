# 统一用户管理最小可行方案

## 背景

- 系统A需要实现用户认知，即知道当前用户是谁、具备哪些基础信息和角色。
- 系统B需要实现用户认证，即完成登录、校验身份、维护登录态。
- 当前不希望每个系统各自实现一套用户管理能力，避免重复建设、账号割裂、维护成本高。

## 目标

- 只建设一套统一用户管理能力。
- 系统A、系统B都接入同一个身份源。
- 先满足登录、用户识别、基础权限信息透传这几个核心能力。
- 方案优先最小可行，后续再逐步扩展单点登录、细粒度权限、组织架构等能力。

## 最小可行方案

采用一个独立的 `IAM` 服务作为统一用户中心，职责尽量收敛：

- 统一维护用户账号。
- 统一维护密码或其他登录凭证。
- 统一提供认证接口。
- 统一签发登录态令牌，例如 `JWT` 或服务端 `session`。
- 统一提供用户基础信息查询接口。

系统职责拆分如下：

### IAM 服务

- 负责用户注册、创建、禁用、重置密码。
- 负责登录认证。
- 负责签发令牌。
- 负责返回用户基础信息，例如 `userId`、`username`、`displayName`、`roles`。

### 系统A

- 不再维护自己的用户表，只保存业务必要的用户映射信息。
- 通过 IAM 签发的令牌识别当前用户。
- 如果需要展示用户信息，调用 IAM 用户信息接口，或从令牌中读取基础字段。

### 系统B

- 不再自己实现账号密码体系。
- 登录页面直接调用 IAM 认证接口，或者跳转到 IAM 统一登录页。
- 登录成功后，使用 IAM 返回的令牌访问本系统接口。

## 推荐架构

```text
用户 -> 系统A/系统B -> IAM

IAM 提供：
1. 登录接口 /login
2. 用户信息接口 /me
3. 用户管理接口 /users
4. 令牌签发与校验能力
```

推荐交互方式：

1. 用户在系统B发起登录。
2. 系统B调用 IAM 认证接口。
3. IAM 校验用户名密码后返回令牌。
4. 系统B 后续请求携带令牌。
5. 系统A 如果也需要识别该用户，直接信任并校验同一套令牌。
6. 系统A、系统B 如需用户详情，统一调用 IAM 的 `/me` 或 `/users/{id}` 接口。

## 为什么这是最小可行方案

- 只新增一个 IAM 服务，不要求系统A、系统B做大规模重构。
- 用户、密码、登录态只保留一份，避免多套账号体系。
- 系统A关注“识别用户”，系统B关注“接入认证结果”，职责清晰。
- 后续如果要升级为单点登录，只需在 IAM 层扩展，不需要每个业务系统重复改造。

## 数据设计建议

最小只需要一张统一用户表，必要时补一张角色关系表。权限不建议一开始全部收口到 IAM，而是采用分层处理。

### users

- `id`
- `username`
- `password_hash`
- `display_name`
- `status`
- `created_at`
- `updated_at`

### user_roles

- `user_id`
- `role_code`

### role_permissions

- `role_code`
- `permission_code`

说明：

- 如果当前阶段权限要求不复杂，可以先只在令牌里放一个 `roles` 数组。
- 系统A、系统B 内部只保留业务侧自己的数据，不再单独维护账号密码。

## 权限怎么处理

推荐采用“统一身份，分层授权”的最小方案。

### 1. IAM 管什么

IAM 只管理跨系统都通用的权限信息：

- 用户属于哪些角色。
- 用户是否启用、是否可登录。
- 一些全局通用身份，例如 `super_admin`、`ops`、`auditor`。

IAM 不建议在第一阶段承接所有业务按钮、菜单、数据范围权限，否则很容易把 IAM 做成大而重的权限平台。

### 2. 业务系统管什么

系统A、系统B继续管理各自的业务权限，例如：

- 哪些菜单可见。
- 哪些接口可调用。
- 哪些按钮可操作。
- 哪些数据范围可访问。

也就是说：

- IAM 负责回答“你是谁”。
- 业务系统负责回答“你在我这里能做什么”。

### 3. 最小落地方式

令牌里只放少量稳定字段：

- `userId`
- `username`
- `roles`
- `status`

业务系统收到令牌后，按下面方式处理：

1. 先校验令牌是否合法。
2. 读取 `userId` 和 `roles`。
3. 将 `roles` 映射为本系统自己的权限集合。
4. 按权限集合控制接口、菜单和按钮。

例如：

- IAM 中用户角色为 `sales_admin`。
- 系统A把 `sales_admin` 映射为 `customer.read`、`customer.edit`。
- 系统B把 `sales_admin` 映射为 `order.read`、`order.approve`。

这样同一个角色可以在不同系统中映射为不同权限，不需要 IAM 知道每个系统的全部权限细节。

### 4. 为什么这样最合适

- 最省成本，先把统一登录和统一用户打通。
- 不会因为权限模型复杂，拖慢 IAM 落地。
- 各系统保留业务自主性，不需要为统一平台牺牲现有权限设计。
- 后续如果多个系统的权限模型逐步趋同，再考虑把角色权限进一步收口。

### 5. 如果要再往前走一步

如果当前就希望做轻量统一授权，可以增加一层“全局角色 + 系统内权限编码”的模型：

- IAM 维护全局角色，例如 `admin`、`operator`、`viewer`。
- 每个系统维护自己的权限编码，例如 `a.customer.read`、`b.order.approve`。
- IAM 只维护角色到权限编码的映射，不管理页面和按钮细节。

这种做法比完全分散更统一，但实施成本会高于最小方案。

### 6. 推荐结论

当前阶段建议：

- 认证统一到 IAM。
- 用户和角色统一到 IAM。
- 权限判断仍放在系统A、系统B内部。
- 系统通过 `roles` 或少量 `permission codes` 做本地授权。

这是“最小可行”且风险最低的权限处理方式。

## 接口最小集合

### 1. 登录

`POST /login`

请求：

```json
{
  "username": "alice",
  "password": "******"
}
```

响应：

```json
{
  "accessToken": "xxx",
  "tokenType": "Bearer",
  "expiresIn": 7200,
  "user": {
    "id": "u_001",
    "username": "alice",
    "displayName": "Alice",
    "roles": ["admin"]
  }
}
```

### 2. 获取当前用户信息

`GET /me`

用于系统A、系统B在拿到令牌后查询当前登录用户。

### 3. 用户管理

- `POST /users` 创建用户
- `GET /users/{id}` 查询用户
- `PUT /users/{id}/status` 启用或禁用用户
- `PUT /users/{id}/password` 重置密码

### 4. 角色与权限

- `GET /users/{id}/roles` 查询用户角色
- `PUT /users/{id}/roles` 维护用户角色
- `GET /roles/{roleCode}/permissions` 查询角色权限

## 落地建议

### 第一阶段

- 建一个独立 IAM 服务。
- 先提供 `login`、`me`、`users` 这几类基础接口。
- 系统B 改为调用 IAM 登录。
- 系统A 改为校验 IAM 令牌并读取用户信息。

### 第二阶段

- 把系统A、系统B原有用户数据迁移到 IAM。
- 清理两个系统内部重复的账号密码逻辑。

### 第三阶段

- 视需要再扩展统一登录页、单点登录、组织架构、细粒度权限控制。

## 需要明确的边界

- IAM 只做身份与基础账号管理，不承载各业务系统自己的业务资料。
- 各系统仍然可以维护自己的业务用户扩展信息，但主身份 `userId` 必须来自 IAM。
- 认证统一收口到 IAM，授权可以先做粗粒度角色透传，后续再细化。
- 如果权限模型复杂，优先把权限规则留在业务系统，避免 IAM 过早变重。

## 结论

最小可行方案是：

- 建立一个独立 IAM 服务作为统一用户中心。
- 由 IAM 统一管理用户、密码、认证和令牌。
- 系统A、系统B都接入 IAM，不再各自维护一套账号体系。
- 先实现统一认证和用户识别，权限与单点登录后续按需增强。

这个方案实现成本最低，且能较好地避免重复建设，适合作为当前阶段的落地方案。
